# 工作流执行约束原理

## 背景（2026-05-07 事件）

### 事件经过

执行"通用漏洞扫描"工作流时，主 AI 在分析爆破测试结果时发生越权行为：

1. **自主诊断**：发现 result_cache.json 有 0 条登录尝试，自主推断"可能是字典文件缺失"
2. **错误预设**：假设字典文件名为 `password.txt`，实际配置为 `passwd.txt`
3. **添加验证步骤**：WORKFLOW.md 未要求验证字典文件，但主 AI 自主添加了验证逻辑
4. **添加 timeout 参数**：在 terminal 调用中添加了 timeout 参数（工作流设计中无此概念）

### 根因分析

workflow-manager SKILL.md 中的"遇到问题时的处理流程"给了主 AI 诊断权限：

```
步骤执行失败
     ↓
[1] 诊断问题
     - 检查文件是否存在、格式是否正确
     - 检查网络连接、权限、依赖服务
     - 查看错误日志
     ↓
[2] 尝试修复（最多3次）
```

这导致主 AI 在遇到问题时自主诊断，而非停止并上报。

### 用户纠正

> "工作流严格按照流程执行，遇到问题就终止。不允许自己添加、判断、修改、诊断执行流程。遇到问题必须上报。诊断是守护 Agent 来处理，不是主 Agent 来处理。"

---

## 解决方案

### 1. 添加"核心约束（最高优先级）"章节

明确主 AI 职责边界：
- 允许：按 WORKFLOW.md 执行、验证输出、上报异常
- 禁止：添加步骤、修改命令、添加参数、自主诊断、自主修复

### 2. 写死执行命令约束

```yaml
WORKFLOW.md 定义: uv run main.py
AI 执行: uv run main.py ✅
AI 执行: uv run main.py --timeout 300 ❌ 禁止添加参数
AI 执行: timeout 300 uv run main.py ❌ 禁止添加 timeout
```

### 3. 写死 terminal 调用约束

```python
# ❌ 禁止
terminal(command="uv run main.py", timeout=600)

# ✅ 正确
terminal(command="uv run main.py")
```

### 4. 职责分离

| 职责 | 主 AI | 守护 Agent |
|------|-------|------------|
| 执行工作流步骤 | ✅ | ❌ |
| 上报异常现象 | ✅ | ❌ |
| 诊断问题原因 | ❌ | ✅ |
| 尝试修复恢复 | ❌ | ✅ |

### 5. 问题处理流程

```
主 AI 发现问题 → 立即停止 → 上报现象 → 等待用户指示
     ↓（用户授权）
守护 Agent 诊断 → 尝试修复 → 通知用户
```

---

## 关键原则

1. **执行 ≠ 诊断**：主 AI 只执行，不诊断
2. **现象 ≠ 原因**：上报异常现象，不推断原因
3. **等待 ≠ 跳过**：等待用户指示，不自主跳过
4. **WORKFLOW.md = 唯一真理**：不添加、不修改、不假设
