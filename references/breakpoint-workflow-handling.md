# 断点工作流在父工作流中的处理模式

## 问题场景

当父工作流调用断点工作流时，断点工作流会在某个步骤返回控制权，后续由心跳监测接管。

**问题**：父工作流应该如何处理断点工作流的返回？

## 正确理解

### 断点工作流的本质

断点工作流（如 home漏扫）是一种**长时间运行的工作流**：
- 步骤分为两部分：同步执行部分（0-5.5）+ 心跳执行部分（6-9）
- 同步执行完成后返回控制权，心跳监测继续在后台执行
- 心跳执行部分可能需要数小时甚至数天

**⚠️ 关键理解（用户纠正 2026-05-12）**：
- 断点工作流执行到断点步骤后，工作流**结束**（不是暂停）
- 后续由心跳监测自动接管，无需等待
- 父工作流可以立即继续执行下一步

### 工作流串行执行约束

根据 memory 中记录的约束：
> **工作流串行执行约束**：
> - 串行工作流必须等待上一步完全执行完毕才能执行下一步
> - 禁止越权执行：不能因为命令超时/卡住就跳过
> - **断点工作流除外**：断点工作流返回后即视为完成

**用户强调（2026-05-12）**：
> "串行就是串行，除了断点工作流，必须是一个节点一个节点完成"

### 正确处理方式

**断点工作流返回后，父工作流立即继续执行下一步（不等待心跳完成）。**

## 实践建议

### 方案：断点返回后立即继续

当断点工作流返回时：
1. 检查工作流类型（是否为断点工作流）
2. 如果是断点工作流，记录心跳信息
3. 立即继续执行父工作流的下一步

**示例**：
```json
{
  "status": "completed",
  "note": "断点工作流已返回，心跳已启动",
  "current_step": "home漏扫",
  "next_step": "爆破测试",
  "heartbeat_job_id": "9d8f2df68f65",
  "heartbeat_note": "后台运行中，无需等待"
}
```

## 错误模式

### ❌ 错误：等待心跳完成

```python
# 错误：断点工作流返回后，等待心跳完成
result = delegate_task("执行 home漏扫")  # 返回断点
# ❌ 等待心跳完成
wait_for_heartbeat_completion()  # 错误：不需要等待
result = delegate_task("执行 爆破测试")
```

**后果**：不必要的等待，浪费用户时间。

### ✅ 正确：断点返回后立即继续

```python
# 正确：断点工作流返回后，立即继续
result = delegate_task("执行 home漏扫")  # 返回断点
# ✅ 立即执行下一步（心跳在后台运行）
result = delegate_task("执行 爆破测试")
```

## 判断断点工作流的方法

检查 _index.yaml 中的节点类型：
```yaml
nodes:
  - id: step_2
    name: 断点返回（启动心跳监测）
    type: breakpoint  # ← 断点标识
    heartbeat:
      enabled: true
```

**判断逻辑**：
- `type: breakpoint` → 断点工作流
- `heartbeat.enabled: true` → 有心跳监测
- 有这些特征的子工作流 → 父工作流**不等待**，立即继续
