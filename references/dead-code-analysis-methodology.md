# 死代码分析方法论

## 概述

本文档记录了工作流技能死代码分析的完整方法论，供未来分析其他技能时参考。

## 分析流程

### 1. 代码扫描

```bash
# 列出所有代码文件
find src -name "*.py" -type f | sort

# 统计代码行数
find src -name "*.py" -type f -exec wc -l {} + | tail -1
```

### 2. 模块调用分析

```bash
# 检查模块是否被外部调用
grep -r "from <module>\|import <module>" --include="*.py" | grep -v "^./<module>/"
```

### 3. 函数调用分析

```bash
# 检查函数是否被调用
grep -n "<function_name>" <file>.py | grep -v "def <function_name>"
```

### 4. 一致性验证

| 检查项 | 方法 |
|--------|------|
| 字段名一致性 | 代码返回值 vs 文档描述 |
| 调用链完整性 | 定义 → 导入 → 调用 |
| 功能可用性 | 入口存在 → 触发路径存在 |

## 根因分类

| 根因类型 | 特征 | 解决方案 |
|----------|------|----------|
| 架构变更遗留 | 代码注释说明"不在此启动" | 删除旧代码 |
| 集成不完整 | 有命令行入口，无技能触发 | 添加自然语言触发 |
| 调用链断裂 | 类实例化但方法未调用 | 修复调用链 |
| 方法未调用 | 定义但无调用 | 删除或修复 |

## 关键教训

1. **先分析代码，再阅读文档**：从代码行为反推文档描述
2. **检查架构注释**：代码中的注释可能说明设计变更
3. **验证调用链完整性**：定义 → 导入 → 调用，每一步都要验证
4. **区分"有入口"和"可触发"**：有命令行入口 ≠ 可通过自然语言触发
