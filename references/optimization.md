# 工作流优化建议规范

## 概述

工作流优化建议是**异步生成**、**用户确认**、**可回退**的优化机制。

## 核心原则

| 原则 | 说明 |
|------|------|
| **建议而非自动** | 生成建议，用户确认后才优化 |
| **异步生成** | 工作流结束后后台分析，不阻塞 |
| **查询时标注** | 查询工作流时提示有优化建议 |
| **可回退** | 优化后可恢复上一版本 |

---

## 触发方式

| 触发词 | 行为 |
|--------|------|
| "优化工作流" | 生成/查看优化建议 |
| "工作流总结" | 生成执行总结 + 优化建议 |
| "查看优化建议" | 查看待处理的建议 |
| "应用建议" | 应用指定建议 |
| "回退工作流" | 恢复上一版本 |

---

## 数据结构

```
workflows/
└── {workflow_name}/
    ├── WORKFLOW.md                    # 当前版本
    ├── WORKFLOW.md.bak                # 上一个版本（回退用）
    ├── status.json                    # 执行状态
    ├── history/                       # 执行历史
    └── optimization_suggestions.md    # 优化建议 ⭐
```

---

## 优化流程

### Phase 1: 生成建议（异步）

```
工作流执行结束
     ↓
收集执行数据
  - status.json（成功/失败）
  - history/（历史趋势）
  - guardian_reports/（问题记录）
     ↓
AI 分析
  - 识别问题
  - 分析根因
  - 生成建议
     ↓
写入 optimization_suggestions.md
```

### Phase 2: 查看建议

```
用户查询工作流
     ↓
检查是否有 optimization_suggestions.md
     ↓
标注：⚠️ 有 N 条优化建议
     ↓
用户点击查看
     ↓
展示建议详情
```

### Phase 3: 应用建议

```
用户确认应用
     ↓
备份当前版本 → WORKFLOW.md.bak
     ↓
应用优化修改
     ↓
更新建议状态
     ↓
通知用户完成
```

### Phase 4: 回退

```
用户请求回退
     ↓
检查 WORKFLOW.md.bak 是否存在
     ↓
恢复备份版本
     ↓
将当前版本保存为 WORKFLOW.md.failed
     ↓
通知用户回退完成
```

## 风险级别

| 级别 | 标识 | 示例 | 处理 |
|------|------|------|------|
| 低风险 | ✅ | 修改超时参数 | 可快速应用 |
| 中风险 | ⚠️ | 修改命令逻辑 | 需仔细确认 |
| 高风险 | 🔴 | 重构步骤结构 | 需测试后应用 |

---

## 建议类型

| 类型 | 说明 | 示例 |
|------|------|------|
| 稳定性优化 | 提高成功率 | 增加重试、超时参数 |
| 性能优化 | 提高执行速度 | 并行化、缓存 |
| 健壮性优化 | 增强容错 | 错误处理、断点续传 |
| 可维护性优化 | 提高可读性 | 步骤拆分、注释 |

## AI 分析维度

### 1. 问题识别

```markdown
分析执行数据：
- 哪些步骤失败率最高？
- 哪些步骤执行时间最长？
- 哪些步骤经常重试？
- 哪些步骤经常卡住？
```

### 2. 根因分析

```markdown
深入分析原因：
- 网络问题？→ 增加超时、重试
- 参数问题？→ 调整参数默认值
- 依赖问题？→ 检查前置步骤
- 资源问题？→ 增加资源限制
```

### 3. 建议生成

```markdown
生成优化建议：
- 具体修改内容（diff 格式）
- 预期收益（可量化）
- 风险评估
- 回退方案
```

## 交互示例

### 示例 1: 查看工作流

```
用户: 查看工作流 data-collection-workflow

助手:
📋 工作流: data-collection-workflow
⚠️ 有 3 条优化建议待处理

状态: ✅ 空闲
最后执行: 2026-04-11 04:00 (成功)
成功率: 80% (最近 5 次)
平均时长: 15 分钟

[查看优化建议] [执行工作流] [查看历史]
```

### 示例 2: 查看建议

```
用户: 查看优化建议

助手:
## 💡 优化建议 (3 条)

### 建议 1: 增加超时参数 ✅ 低风险
步骤 3 经常超时，建议增加 --max-time 参数
预期收益: 成功率 ↑ 20%

### 建议 2: 添加分页机制 ⚠️ 中风险
数据量大时内存溢出，建议分页获取
预期收益: 稳定性 ↑ 30%

### 建议 3: 优化重试策略 ✅ 低风险
重试间隔太短，建议使用指数退避
预期收益: 减少无效重试

[应用建议 1] [应用建议 2] [应用建议 3] [全部应用] [重新制定]
```

### 示例 3: 应用建议

```
用户: 应用建议 1

助手:
正在应用建议 1: 增加超时参数...

✅ 已备份当前版本
✅ 已更新 WORKFLOW.md
✅ 已更新优化建议状态

建议 1 已应用。如不满意可回退：
"回退工作流 data-collection-workflow"
```

### 示例 4: 回退

```
用户: 回退工作流 data-collection-workflow

助手:
正在回退...

✅ 已恢复到上一版本
✅ 当前版本已保存为 WORKFLOW.md.failed

回退完成。是否重新制定优化方案？
```

### 示例 5: 重新制定

```
用户: 重新制定优化方案

助手:
好的，请告诉我之前方案有什么问题？

用户: 建议 2 的分页参数不对，应该是 50 不是 100

助手:
明白了，我来重新制定建议 2...

## 新的建议 2: 添加分页机制 ⚠️ 中风险

**修改**:
```diff
- curl http://api/data
+ curl http://api/data?page_size=50
```

**预期收益**: 稳定性 ↑ 30%

确认应用新方案？
```

## 与其他功能的关系

| 功能 | 关系 |
|------|------|
| 执行工作流 | 执行结束后触发建议生成 |
| 查看工作流 | 标注是否有优化建议 |

## 配置项

```yaml
optimization:
  enabled: true
  auto_generate: true        # 执行结束后自动生成
  notify_on_suggestions: true # 有新建议时通知
  max_suggestions: 5         # 最多保留建议数
  history_retention: 30      # 优化历史保留天数
```
